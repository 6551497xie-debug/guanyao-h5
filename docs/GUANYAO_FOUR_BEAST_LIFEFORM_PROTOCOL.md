# GUANYAO Starbeast Life Archetype Protocol
# 观爻星兽生命原型协议

版本：V2.0

状态：ACTIVE STARBEAST SEMANTIC PROTOCOL

上位语义源：[GUANYAO_WORLDVIEW_CONSTITUTION_V2.md](./GUANYAO_WORLDVIEW_CONSTITUTION_V2.md)

## 00｜协议定位

本协议依据《GUANYAO 世界观宪法 V2.0》，校准观爻中星兽的哲学定位、计算边界、产品关系、视觉气质与工程边界。

本协议不是新的推导引擎，不改变出生日期、农历、二十八宿、四象兽或母码的既有计算结果；也不新增 Runtime 状态、路由、存储字段或卡面。

本协议正式锁定：

> 星兽 = 生命原型（Life Archetype）

“四象兽”是青龙、朱雀、白虎、玄武这一传统四象形态的集合名称；“星兽”是用户在观爻中的生命关系名称。工程字段可继续使用 `fourSymbol` 与 `starbeast`，但前台必须优先让用户感到“星兽就是我”，而不是“系统给了我一个角色”。

## 01｜星兽是什么

星兽是：

> 从宇宙时间中诞生，携带自身生命密码，来到现实世界体验、成长和变化的另一个自己。

它不是为了证明你是谁，而是陪你经历你是谁。

星兽不是独立于用户之外的生命。它是用户自身生命力量被看见后形成的生命原型。

它承载：

- 原始生命倾向；
- 感受世界的方式；
- 面对现实力量时的生命姿态；
- 在经历中留下的变化痕迹。

星兽承担三项产品作用：

1. 让用户第一次感到“有一个生命一直与我同行”。
2. 让身体、情绪、思想、行动、记忆与动机中的体验获得可感知的生命形态。
3. 让一次经历留下的不只是答案，而是一枚可以回望的生命印记。

## 02｜星兽不是什么

星兽不是：

- 人格标签；
- 心理分类；
- 性格测试结果；
- AI 生成角色；
- 外在守护者或救世主；
- 宠物、召唤兽或游戏战宠；
- 命运预测物；
- 吉凶判断；
- 用户需要照顾的负担；
- 积分、等级、奖励或惩罚系统；
- 商品化实体按钮。

星兽永不责备、评分、羞辱或惩罚用户。

用户没有完成某一步时，星兽不得表现为受伤、失望、衰弱或责怪。星兽的“护持”是用户生命自身的承接姿态，不是外部英雄替用户解决人生。

## 03｜生命原型稳定，生命姿态变化

星兽的四象归属来自出生日期，是稳定的生命原型锚点。

稳定不等于标签。

青龙、朱雀、白虎、玄武不规定用户一生只能是什么样的人，也不预测用户会拥有怎样的命运。它们描述生命最初携带的一种原型姿态。

真正变化的是星兽进入现实之后的状态：

- 它如何感受环境力量；
- 它在六维生命界面中如何收缩、展开、停住或前行；
- 它如何在觉察之后重新选择方向；
- 它在这一次经历中留下什么生命印记。

因此，观爻不再使用“用户不是永远属于某只兽”来表达反标签立场。

正确表达是：

> 你的生命原型有稳定的来处，但你不会被它限制；星兽的姿态会与你一起经历和变化。

## 04｜与传统四象的关系

观爻继承青龙、朱雀、白虎、玄武的天文文化来源与生命意象，不复刻神兽崇拜，也不把四象改造成玄学召唤系统。

四象在观爻中表达四种原始生命姿态：

- 青龙：生发、游走、打开路径；
- 朱雀：显明、表达、上升与照见；
- 白虎：边界、清醒、守护与决断；
- 玄武：沉潜、承载、护持与回收。

这些词不是人格定论，而是星兽生命力量的姿态语言。

白虎尤其不得被简化为攻击、征服或战斗。它的力量来自边界感、安静的威严与必要时的决断。

## 05｜星兽推导链

星兽的正式推导链保持不变：

```text
用户输入公历出生日期
→ 转换为农历月、日
→ 查询二十八宿值日星宿
→ 归入青龙 / 朱雀 / 白虎 / 玄武
→ starbeast.fourSymbol
```

计算边界锁定如下：

1. 星兽只取决于出生日期这一时间维度。
2. 出生地点不参与星兽推导。
3. 出生时辰不参与星兽推导。
4. 同一公历出生日期在同一历法协议下得到相同星兽。
5. 地理信息只表达出生地点上下文，不承载正式四象兽语义。
6. 正式消费者优先读取 `starbeast.fourSymbol`；旧字段只能作为历史数据 fallback。

这一推导是一套可重复的文化符号映射，不是对现代天文学因果、超自然力量或终身命运的证明。

## 06｜星兽与母码相互独立

星兽不推导八卦，四象归属也不是母码的输入。

母码生成链保持不变：

```text
用户输入公历出生日期与时辰
→ 转换为农历年、月、日
→ 地支时辰换算为序数
→ 农历年数 + 月数 + 日数 + 地支时数
→ 卦以八除
→ 先天八卦 / 母码
```

两条链的关系是：

```text
出生日期 → 星宿 → 四象 → 星兽生命原型

农历年月日 + 地支时数 → 八除 → 母码原始生命倾向
```

星兽结果中的方向与 `symbolicTrigram` 只保留传统四象文化对应，不得成为母码 trigram source。

星兽与母码可以在视觉上“入卦”合流，但不得在计算上互相伪造来源。

## 07｜星兽与产品主链

世界观宪法 V2.0 下，星兽贯穿的不是人格资产生产线，而是一条生命体验链：

```text
星兽携带生命原型来到现实
→ 母码呈现原始生命倾向
→ 压力种子带来环境力量
→ 六维承接真实生命体验
→ Gravity 显出现实引力
→ Choice 重新选择生命方向
→ Crystal 留下生命印记
```

这条世界观解释不改变当前工程因果顺序：

```text
母码
→ 压力种子成局
→ 六维传导
→ ChangeExperience / PersonaMigrationImpact
→ Crystal
```

工程中的 `persona`、`asset`、`ring` 等既有名称可作为兼容标识继续存在，但不得反向把星兽定义为人格分类器或资产生成角色。

## 08｜星兽与母码

母码不再只解释为人格底盘。

母码是：

> 星兽来到现实世界时携带的原始生命倾向。

星兽提供生命形态，八卦提供变化结构。两者在母码卡上合流，表达“这一种生命如何带着自己的原始倾向来到世界”。

后台可以使用“四象兽 × 八卦”描述视觉组合；前台可以使用“入”的语言：

- 朱雀入艮；
- 青龙入震；
- 白虎入兑；
- 玄武入坎。

“入”只表示生命形态进入母码视觉结构，不表示四象兽参与母码计算。

母码是出发点，不是测试结果，也不是终身人格定义。

## 09｜星兽与六维

六维不是对星兽进行六项分析。

六维是星兽通过人体验现实的六个生命界面：

- 身体：现实首先如何被生命感知；
- 情绪：体验如何在内在形成波动；
- 思想：生命如何解释正在发生的现实；
- 行动：生命如何停住、靠近、离开或回应；
- 记忆：过去如何回到此刻；
- 动机：生命真正想守住、寻找或创造什么。

六维中的星兽不是讲解员，也不是评分者。它通过呼吸、姿态、光场和距离变化，让用户看到同一现实力量如何穿过完整的自己。

当前 `/dynamics` 已有六维推进、节点反馈、完成态与 Crystal 承接。星兽动态反馈仍属于后续演进，本协议不要求本刀接入。

## 10｜屏幕出现规则

### ENTRY / 一光入口

只出现生命之光、星点或尚未完整显形的生命轮廓。

用户应感到“它一直在那里等我”，而不是“系统正在生成一只角色”。

### 出生坐标 / 星兽显形

星兽第一次被辨认。这里表达的是来自宇宙时间的生命原型，不是出生地点方向，也不是人格诊断。

### 母码生成与母码卡

星兽与母码在视觉上合流。卡面表达生命原型携带原始生命倾向进入现实，不把两个结果做成标签拼盘。

### 压力种子

环境力量成为当前焦点。星兽不抢叙事，也不替用户消灭压力；它与用户一起进入现实。

### Gravity / 六维

现实引力通过六个生命界面发生。星兽的变化应表现“我正在经历”，而不是“系统正在分析我”。

### Choice

星兽不替用户选择。它只在用户重新取得方向感时，显出生命重新回到自身的姿态。

### Crystal

星兽成为这次经历的来源痕迹。Crystal 是生命印记，卡面只是它的可见载体。

### 人格年轮 Lite

当前名称与存储保持兼容。其世界观含义是生命经历的沉积记录，不是人格评分档案。

## 11｜情绪与关系原则

星兽与用户之间不是“主人与宠物”，也不是“用户与外部守护者”。

它们是同一个生命的两种可见状态。

星兽的在场必须：

- 克制而不冷漠；
- 温暖而不讨好；
- 安静而不空洞；
- 威严而不压迫；
- 古老而不故弄玄虚；
- 孤独但不让用户感到被遗弃。

它不靠安慰话制造依赖，不把用户弱点角色化，也不使用“我替你承受一切”的外部拯救叙事。

正确感受是：

> 不是系统在评价我，而是我生命中的某种感受终于被我看见。

## 12｜视觉宪法

星兽的第一视觉目标不是“酷”，而是“生命在场”。

所有四象共同遵守五种气质：

- 威严；
- 孤独；
- 温暖；
- 古老；
- 安静。

允许通过星点、呼吸、停驻、回光、距离与光场变化表达生命状态。

禁止：

- 宠物养成感；
- 神兽召唤感；
- 游戏战宠感；
- 过度攻击与力量炫耀；
- AI 随机生成角色感；
- 超级英雄拯救感；
- 因用户未完成而受伤或变暗；
- 用红色失败态、能量不足或等级变化刺激焦虑；
- 用高噪声特效替代生命关系。

白虎必须尤其避免獠牙、扑杀、征服和战斗胜利叙事。它应像一个从星空走来的古老生命：清醒、安静、有边界，也有温度。

## 13｜动作语言

星兽动作应描述生命状态，而不是技能释放：

- 显形：生命从星点中被辨认；
- 停住：重新感到边界；
- 护持：生命自身围住重要之物；
- 开路：为新的方向留出空间；
- 收束：从外界牵引回到自身；
- 张线：让经历之间的关系可见；
- 回光：在觉察后重新接回一点自己；
- 沉积：让一次经历留下生命印记。

四象动作倾向：

- 青龙：游走 / 生发 / 开路；
- 朱雀：展翼 / 显明 / 照见；
- 白虎：伏身 / 守界 / 决断；
- 玄武：盘伏 / 承载 / 护核。

这些动作不得包装成攻击技能、任务奖励或数值成长。

## 14｜Crystal 与沉积

每一局结束后，星兽不应像完成任务的角色一样退出。

它在这一局经历中的变化，沉积为 Crystal：

> 一次生命体验留下的星辰印记。

卦码卡、人格年轮 Lite、星点或未来兽痕都只是承载生命印记的媒介。

它们不得把 Crystal 表达为：

- 成功证明；
- 完成奖励；
- 人格评级；
- 稀有度竞争；
- 命运结论。

## 15｜1.0 当前实现边界

观爻 1.0 当前保持：

- 星兽由正式出生日期链路确定；
- 正式四象兽语义归入 `starbeast.fourSymbol`；
- 星兽与母码分别推导，在视觉层合流；
- 星兽在出生坐标与母码阶段显形；
- `/dynamics` 保留现有六维反馈与 Crystal 链；
- 人格年轮 Lite 保留现有星点沉积；
- 允许静态视觉资产与现有 fallback；
- 不做运行时 AI 生图；
- 不新增重型 3D 或 WebGL 依赖。

本刀不要求：

- 重做现有卡面；
- 接入星兽六维动画；
- 修改星兽推导结果；
- 修改母码结果；
- 改变用户存量数据；
- 新增 Choice Runtime；
- 新增 384 爻向；
- 扩展商业化与收藏系统。

## 16｜后续演进原则

后续星兽演进必须按以下顺序进行：

1. 先校准前台语言，使用户感到“星兽就是我”。
2. 再校准视觉气质，去除战宠、召唤与 AI 角色感。
3. 再建立六维中的生命姿态反馈。
4. 最后才考虑兽痕、动态沉积或更完整的生命轨迹表达。

动态能力越强，越要守住人的主体性。星兽不能因拟人化而变成新的外部权威或情感依赖对象。

## 17｜工程边界

本协议校准只修改文档语义，不改变：

- `guanyaoStarbeastEngineService`；
- `guanyaoBirthCalendarService`；
- `guanyaoLunarTrigramLandingResolver`；
- 母码生成与八除规则；
- `GravityPage.tsx`；
- Runtime / service / types / mocks；
- localStorage schema；
- route；

## 19｜最终定义

星兽不是系统生成给用户的另一个角色。

星兽是用户终于能够看见的自己。

它有稳定的生命来处，也会在每一次真实经历中改变姿态。

它不告诉用户应该成为谁。

它陪用户看见：

> 我如何来到这里，我正在经历什么，我仍然可以选择怎样继续成为自己。
