# GUANYAO Four Beast Dynamic Lifeform Protocol
# 观爻四象兽动态生命体协议

## 00. 协议定位

星座把人格固定成一个标签。
生肖把人格固定成一个动物。
观爻把人格放回现实压力和六维人格空间中，看见它如何变化。

本协议不是视觉规范，不是神兽设定集，也不是宠物系统方案。它是观爻 1.0 之后所有四象兽、光兽、母码卡、卦码卡、本局结晶、人格年轮中生命体表达的上位协议。

观爻不是另一个星座系统，也不是东方生肖卡。观爻借鉴星座与生肖的人格具象化能力，但核心不是标签，而是变化。

四象兽不是“你是哪只兽”，而是：

这一局，你的光如何成兽、入卦、遇压、传导、结晶。

## 01. 四象兽是什么

四象兽是用户原始星光在人格结构中聚成的动态生命形态。

它由星点聚成。四象决定生命姿态。八卦决定人格骨架。现实压力激活动作。六维人格空间让它逐层显影。本局结晶留下痕迹。人格年轮沉积星点与兽痕。

四象兽不是外来的宠物或神兽。它其实就是用户自己，是用户某种底层力量在这一局现实压力中的生命化显影。

在观爻中，四象兽承担三件事：

- 让抽象人格结构有一个可认领的生命形象。
- 让现实压力下的反应不再只是冷数据，而是可被看见的姿态。
- 让一局完成后的结晶不只是一段结果，而是一点可以沉积的人格痕迹。

## 02. 四象兽不是什么

四象兽不是传统神兽复刻。不是玄学预测物。不是宠物系统。不是用户负担。不是惩罚系统。不是积分养成物。不是商业化实体按钮。不是 3D 琉璃卡。不是 384 爻器。不是 archive 或 collectible asset system。

星兽永不惩罚用户。压力不是用户罪责。四象兽不责备、不评分、不羞辱用户。

四象兽不应该说“你失败了”。它也不应该因为用户没有完成某一步而变暗、失望、受伤或责怪用户。它只负责在场、承接、显影、护持和留下痕迹。

## 03. 与星座 / 生肖的关系

西方星座把人格具象成星图标签。中国生肖把人格具象成动物年轮。它们之所以流行，是因为把抽象人格变成可命名、可认领、可传播的生命形象。

观爻吸收的是这种“人格具象化机制”，不是复制它们的固定标签逻辑。

观爻的差异在于变化：同一束光，在不同现实压力和六维人格空间中，会显出不同姿态。用户不是被归类成一个固定标签，而是在一局现实压力里看见自己的反应如何被激活、传导、转化并结晶。

## 04. 与传统四象的关系

观爻借用青龙、朱雀、白虎、玄武的方位与姿态原型，但不复刻传统神兽崇拜，也不做玄学神兽召唤。

四象兽在观爻中是人格生命姿态：

- 青龙：生发、游走、打开路径。
- 朱雀：显明、表达、上升与照见。
- 白虎：边界、锋利、守护与切断。
- 玄武：沉潜、护持、承压与回收。

这些名字不是命定身份，而是人格力量在方位、时序、压力和六维传导中的姿态语言。

## 05. 星光来源：从星宿到原始人格

古典天象体系中，人常被理解为“禀星气而生”。观爻不采用命定论，也不宣称星宿决定终身命运。

观爻将其现代转译为：用户携带某种原始光质进入现实。

因此，原始坐标不是普通资料填写，而是校准原始光位。光兽不是外来的角色，而是那束星光在用户身上聚成的生命形态。母码不是测试结果，而是原始人格第一次成形的样子。

观爻不说“你是哪颗星，你就是哪种人”。观爻只问：这一局里，这束光会怎样入卦、遇压、传导、结晶。

## 06. 动态原则：不是标签，而是变化

四象兽不是固定身份。用户不是永远属于某只兽。母码也不是终身标签。

四象兽会在不同现实压力中呈现不同姿态。观爻观察的不是“你是谁”，而是“这一局你如何变化”。

这一局，你的光如何成兽、入卦、遇压、传导、结晶。

这句话是四象兽动态生命体协议的核心句。所有四象兽视觉、母码卡、卦码卡、本局结晶和人格年轮的表达，都必须围绕这一句展开。

## 07. 变化数学：8 卦、64 卦、384 爻向

观爻的变化不是随机表演，而是建立在卦象结构上的人格变化模型。

8 卦：

- 表示人格底层原型。
- 在观爻中表现为 8 母码。

64 卦：

- 表示原始母码遇到现实压力后的本局结构。
- `mother lower trigram + pressure upper trigram = currentHexagramProfile`。

384 爻向：

- 表示未来人格具体走向系统。
- 用于记录一局中更细的动作位置、变化切口和人格走向。
- 观爻 1.0 不接入。

观爻 1.0 已完成 8 母码与 64 本局卦码的显影闭环，并已完成六维反馈雏形。观爻 1.0 未完成 384 爻向。

1.0 不接 `yao-device`，不接 384 爻器，不接旧 R8 页面流。

## 08. 四象兽 × 八卦 = 母码形象

四象兽提供生命姿态。八卦提供人格骨架。四象兽 × 八卦生成母码形象。

同一个八卦，因四象兽不同，主视觉不同。因此母码内容是 8 套，但主视觉资产应该支持 4 × 8 = 32 组。

后台可以使用“四象兽 × 八卦”表达组合关系。前台更适合使用“入”的语言，例如：

- 朱雀入艮
- 青龙入震
- 白虎入兑
- 玄武入坎

“入”表达生命体进入卦骨架，不是两个标签并列。

示例：

- 朱雀入艮：展翼但定格，被边界收束的火羽。
- 青龙入震：长身启动，向前破开滞点。
- 白虎入兑：锋利收口，切开冲突并保留转圜。
- 玄武入坎：沉入深水，护住压力中的核心。

## 09. 屏幕出现规则

### 1. ENTRY / 光兽入口

生命体只以光兽或光点在场，不提前暴露完整四象兽。入口阶段的目标是让用户感觉“有东西在等我”，不是提前解释完整人格结构。

### 2. 原始坐标 / 方位显影

四象兽第一次显影。它说明用户的方位力量开始聚形，但它还不是正式资产卡。

### 3. 母码生成

四象兽与八卦合流，进入母码形象。此时生命姿态开始进入人格骨架。

### 4. 母码卡正面

四象兽进入卡面主视觉。正面是母码形象识别面，应让用户第一眼看到“我的母码形象已经显影”。

### 5. 母码卡背面

四象兽退为背景或来源，不抢人格原型解码。背面负责让用户读懂“我为什么会这样，以及我可以如何转化”。

### 6. 压力种子轴

四象兽不抢主线，压力成为主角。兽的作用是陪用户带着母码进入现实压力，而不是在压力轴上再次成为视觉中心。

### 7. /dynamics 六维传导

1.0 已有六维反馈雏形。当前以六维推进、当前维度高亮、节点反馈、结晶承接为主。未来可让兽作为六维传导的生命体反馈。

### 8. 本局结晶 / 卦码卡

四象兽应成为本局结晶的来源痕迹，而不是重新抢主角。它在这里的角色是“这一局生命形态曾经如何通过压力留下痕迹”。

### 9. 人格年轮 Lite

1.0 只保存星点。后续可沉积兽痕、四象印记和结晶来源。

## 10. 六维反馈规则

### 六维不是空白传导

观爻 1.0 的 `/dynamics` 已经建立六维反馈雏形：

- 用户每推进一维，当前人格维度会被视觉聚焦。
- 已完成维度会退后为轨道。
- 节点反馈区会给出当前维度的简化反馈。
- 第 6 维完成后会进入候选结晶。
- 如果本局上下文完整，则生成真实 `currentCrystalEndState`。
- 真实结晶进入 MOLD / CARD 两级终点。

这意味着 1.0 的六维传导不是空白占位，而是已经完成了“现实压力进入六维人格空间后逐层显影”的最小产品骨架。

### 六维定义

协议层的六维包括：

- 身体：压力首先进入身体反应。
- 情绪：压力激活情绪波动。
- 思想：压力进入解释、判断和念头。
- 行动：压力推动或阻断具体行为。
- 记忆：压力唤起旧经验、旧伤口和旧模式。
- 动机：压力暴露真正想守住、证明、逃离或靠近的东西。

代码中 UI key 存在 `action` / `goal`，协议层对应“行动 / 动机”。部分前台显示可能仍写“行为 / 目标”，但协议锁定的六维表达是：身体、情绪、思想、行动、记忆、动机。

### 当前实现路径摘要

1.0 的六维反馈由以下结构共同形成：

- `ExecutionSnapshot`：记录当前六维推进状态，包括当前节点、已完成节点和运行状态。
- `enginePhase`：记录运行态是否完成。
- `uiPhase`：记录前台展示阶段。
- `CORE_STAR_BLOOM`：用户点击光点后推进节点。
- `READY_TO_CRYSTALLIZE`：六维完成后的候选结晶态。
- `currentCrystalEndState`：本局上下文完整时生成的真实本局结晶状态。
- MOLD / CARD：真实结晶形成后的两级终点展示。

这些名称属于工程实现，但在产品协议中代表一条清晰路径：压力种子进入六维，用户逐维读取反应，六维完成后进入候选结晶；如果母码、本局卦码与压力上下文完整，则沉积为真实本局结晶。

### 1.0 已完成

- 六维推进。
- 当前维度视觉焦点。
- 节点反馈雏形。
- 第 6 维自动完成。
- 候选结晶态。
- 真实 `currentCrystalEndState` 承接。
- MOLD / CARD 结晶终点。

### 1.0 未完成

- 星兽在六维中的动态动作反馈。
- 每一维的完整差异化人格解码。
- 384 爻向。
- `yao-device`。
- 具体人格走向系统。
- 星兽动作沉积进人格年轮。

### 六维与四象兽关系

当前 1.0 中，四象兽主要在 LaunchLab、原始坐标、母码卡阶段显影。`/dynamics` 中星兽动作尚未成为主反馈，但六维反馈已经为 2.0 留出位置。

未来 2.0 可将星兽动作映射到六维：

- 身体：星兽收缩、伏身、护持。
- 情绪：光场波动、羽光暗亮。
- 思想：张线、聚焦、分叉。
- 行动：破局、停住、前探。
- 记忆：回光、旧痕浮现。
- 动机：星核显影、守护核心。

### 六维与 384 爻向关系

六维回答：压力进入了人格的哪一层。

384 爻向回答：这一层中出现了哪一种具体走向。

384 爻向未来应建立在六维反馈骨架上。1.0 只做到六维反馈雏形，不接 384。

### 白皮书可复用表述

观爻 1.0 的 `/dynamics` 已不是空白传导页。它已经建立了六维反馈雏形：用户每推进一维，当前人格维度会被视觉聚焦，并出现对应节点反馈。它不是完整星兽动态反馈，也不是 384 爻向，但已经建立了“现实压力进入六维人格空间后逐层显影”的产品骨架。

## 11. 情绪锚定规则

四象兽的作用是让用户在冷系统里感到有生命体在场。

它不说安慰话，但通过姿态、光点、停驻、收束、护持来承接用户。它的陪伴是克制的、非侵入式的。它不替代用户判断，也不把用户弱点人格化成负担。

四象兽要让用户感到：

这不是系统在评价我，而是我的某种反应正在被看见。

这也是它与宠物系统的差异：它不需要用户照顾它，不用卖萌，不制造陪伴负担。它只是把用户自己的底层力量以生命体的形式显影出来。

## 12. 入卡规则：母码卡 / 卦码卡

### 母码卡

母码卡正面必须呈现四象兽入卦的形象。图片不应包含文字，文字由前端渲染。

四象兽主视觉要解释“为什么是这个母码”。正面建议结构：

1. 卦名 / 卦符 / 四象兽名
2. 角色名
3. 主视觉区
4. 一句话识别
5. 三枚资产标签

母码卡背面则负责人格原型解码：原型识别、惯性反应、行为代价、转化方向，以及必要的来源说明。

### 卦码卡 / 本局结晶卡

四象兽不再只是母码来源。它应该作为本局压力完成后的痕迹进入卡面。

1.0 可只保留来源语义。1.1 再做兽痕或结晶印。

## 13. 动作语言

基础动作词：

- 显影：从星点聚形。
- 停住：边界收束。
- 护持：围住压力核心。
- 破局：向前打开裂缝。
- 收束：从扩散回到一点光。
- 张线：从兽伸出因果线。
- 回光：完成后留下一点星。
- 沉积：进入人格年轮。

四象动作倾向：

- 青龙：游走 / 生发 / 开路。
- 朱雀：展翼 / 显明 / 定格。
- 白虎：伏身 / 切线 / 守界。
- 玄武：盘伏 / 承压 / 护核。

这些动作语言用于统一未来视觉、动效、文案和结晶沉积表达，避免四象兽在不同屏幕里变成互不相关的装饰图案。

## 14. 沉积规则：本局结晶 / 人格年轮

每一局结束，四象兽不应消失。

1.0 Lite：保存为人格年轮上的一枚星点。

后续：星点可带四象兽来源、母码来源、本局卦码来源。更完整的年轮系统可以继续沉积兽痕、星点、结晶印记，但这不属于 1.0 production 主链路。

当前不扩展完整 archive，不接 collectible asset system，不做 90 天复盘。但协议保留未来“兽痕 / 星点 / 结晶印记”的方向。

## 15. 1.0 最小实现

观爻 1.0 只要求：

- 四象兽真实来自 `originMotherContext` / `motherCodeProfile`。
- 四象兽在原始坐标阶段显影。
- 四象兽进入母码卡正面。
- 母码卡显示四象兽 × 八卦。
- 母码卡正面有四象兽入卦主视觉雏形。
- `/dynamics` 已有六维反馈雏形。
- 本局结晶和人格年轮 Lite 不强行扩展兽痕。
- 不接插件、不接 WebGL、不做运行时 AI 生图。
- 允许静态图片资产作为主视觉，缺失时 fallback 到 canvas 水印。

1.0 的重点是闭环稳定：原始坐标、母码、压力、六维、本局结晶、人格年轮 Lite。四象兽必须服务这个闭环，而不是把 1.0 拉向重资产系统。

## 17. 工程边界

1.0 上线前不接 three.js / WebGL / 大型粒子库。不做运行时 AI 生图。不把图中文字烘焙进图片。

静态图只做主视觉。文案仍由前端渲染。
